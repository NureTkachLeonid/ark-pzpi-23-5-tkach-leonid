1. МЕТА
Освоїти стандарти оформлення коду (Code Conventions) на прикладі системи догляду за рослинами "Smart Plant Care System".

2. ХІД РОБОТИ

2.1. Важливість єдиного стилю кодування для IoT-проєктів
У розробці програмного забезпечення, особливо для IoT-систем, дотримання єдиного стилю кодування є критично важливим. IoT-системи часто складаються з багатьох компонентів (мікроконтролери, сервери, мобільні додатки), написаних різними мовами. Єдиний стиль дозволяє команді розробників швидко орієнтуватися в коді, зменшує час на введення нових учасників у проєкт та знижує ризик виникнення помилок. "Чистий код" (Clean Code) є запорукою легкої підтримки та масштабування системи в майбутньому.

2.2. Правила іменування (Naming Conventions)
У проєкті використовуються наступні правила іменування:
- Змінні та методи: використовується CamelCase. Назва повинна починатися з маленької літери, а кожне наступне слово — з великої.
  Приклад: `soilMoisture` (вологість ґрунту), `getSensorData()` (отримати дані з датчика).
- Класи та інтерфейси: використовується PascalCase. Кожне слово починається з великої літери.
  Приклад: `PlantSensor` (датчик рослини), `WaterPumpController` (контролер водяного насоса).
- Константи: використовується SNAKE_CASE (верхній регістр з розмиканням слів підкресленням).
  Приклад: `MAX_HUMIDITY_THRESHOLD`, `DEFAULT_TIMEOUT_MS`.

2.3. Правила коментування
Код повинен бути самодокументованим, але складні алгоритми та публічні API вимагають пояснень.
- Javadoc/Doxygen: використовується для опису класів та методів. Коментар повинен містити опис того, що робить метод, його параметри (@param) та значення, що повертається (@return).
- Inline коментарі: використовуються рідко, тільки для пояснення неочевидних рішень у бізнес-логіці.

2.4. Принципи SOLID
- Single Responsibility Principle (SRP): Кожен клас має відповідати лише за одну частину функціоналу. Наприклад, клас `SensorReader` відповідає тільки за зчитування даних, а `DataUploader` — за їх відправку на сервер.
- Open/Closed Principle (OCP): Система має бути відкрита для розширення, але закрита для змін. Додавання нового типу датчика не повинно ламати існуючий код.

3. ПРИКЛАДИ КОДУ

3.1. Приклад "Поганого коду"
Нижче наведено приклад коду, який важко читати та підтримувати: змінні не мають змістовних назв, відсутнє форматування, все зліплено в купу.

class pump{
public void go(int a){

3.2. Приклад "Хорошого коду"
Той самий функціонал, але переписаний з використанням стандартів оформлення.

/**
 * Контролер для управління водяним насосом.
 */
public class WaterPumpController {

    private static final int MOISTURE_THRESHOLD = 30;

    /**
     * Запускає полив, якщо вологість нижче порогового значення.
     * @param soilMoisture рівень вологості ґрунту у відсотках.
     */
    public void activateWateringIfNeeded(int soilMoisture) {
        if (soilMoisture < MOISTURE_THRESHOLD) {
            System.out.println("Soil moisture is low. Starting watering process...");
            performWatering();
        } else {
            System.out.println("Soil moisture is optimal.");
        }
    }

    private void performWatering() {
        // Логіка увімкнення насосу
    }
}
